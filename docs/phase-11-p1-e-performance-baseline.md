# 阶段 11 P1-E：单实例性能基线

## 目标

在 Redis Lua 优化前，记录当前单 API 实例、MySQL 串行事务方案的性能和一致性表现，为 P1-F 提供对比基线。

## 执行方式

无需安装 k6。项目提供零新增依赖的 Node.js 压测脚本：

```bash
pnpm perf:baseline
```

脚本会构建项目、临时启动 API、执行压测并自动关闭 API。

## 测试环境

| 项目 | 值 |
|---|---|
| 平台 | macOS `arm64` |
| CPU | Apple M2，`8` 核 |
| 内存 | `16 GB` |
| Node.js | `v24.16.0` |
| API 实例 | `1` |
| 数据库 | 本地 Docker MySQL |
| 当前热点状态 | MySQL 事务，尚未迁移到 Redis Lua |

## 基线结果

| 场景 | 请求数 | 吞吐 | P50 | P95 | 最大延迟 | 状态码分布 | 一致性 |
|---|---:|---:|---:|---:|---:|---|---|
| 单场热点竞拍 | `200` | `984.01 req/s` | `13.91 ms` | `27.44 ms` | `33.54 ms` | `201: 10`、`400: 48`、`409: 142` | 通过 |
| 十场并行竞拍 | `100` | `565.19 req/s` | `16.94 ms` | `20.73 ms` | `28.04 ms` | `201: 92`、`409: 8` | 通过 |
| 重复请求与非法低价 | `100` | `1496.07 req/s` | `45.83 ms` | `62.22 ms` | `63.53 ms` | `201: 50`、`400: 50` | 通过 |
| 封顶成交瞬间竞争 | `30` | `1297.80 req/s` | `15.08 ms` | `21.72 ms` | `23.00 ms` | `201: 1`、`400: 13`、`409: 16` | 通过 |

## 一致性结果

- 单场热点竞拍最终价格严格达到预期值。
- 十场竞拍并行执行后，`10/10` 场最终价格正确。
- `50/50` 个重复请求被识别为幂等请求。
- `50/50` 个非法低价请求被拒绝。
- 封顶价并发竞争仅接受 `1` 个请求。
- 封顶竞拍状态为 `SOLD`，仅生成 `1` 个订单。

## 结论

当前单实例方案能够保证正确性，但热点竞拍存在较多冲突：

- 单场热点竞拍中，`200` 个请求只有 `10` 个按轮次成功，其余请求被规则校验或事务冲突拒绝。
- 十场并行竞拍中仍出现 `8` 个 `409`，说明 MySQL `Serializable` 事务会产生跨场次竞争。
- 当前架构适合演示和低并发业务，但不适合作为高并发最终方案。

## P1-F 优化重点

1. 将竞拍热点状态迁移到 Redis。
2. 使用 Redis Lua 原子执行幂等、状态、最低价、延时和价格更新。
3. 使用 Redis ZSET 保存排行榜，避免每次成功出价聚合 MySQL。
4. MySQL 继续承担出价流水、最终状态和唯一订单兜底。
5. 优化后重新运行同一压测脚本，对比 `409` 数量、吞吐和延迟。

## 当前边界

- 本轮测量的是 REST 出价确认延迟，不包含 WebSocket 广播到达延迟。
- 本轮为本机开发环境基线，不代表生产环境容量。
- 在线连接数和多实例广播留到后续扩展验证。
