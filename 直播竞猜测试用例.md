# 直播竞猜测试用例

> 说明：本文档针对课程实践中的独立直播竞拍系统编写，不面向抖音线上生产系统，也不依赖真实抖音平台接口。  
> 测试目标是从简单到复杂逐步验证系统在竞拍规则、并发一致性、WebSocket 实时同步、断线重连、接口安全、性能压测等方面是否存在问题，从而为后续系统优化提供依据。

---

## 1. 测试范围与测试目标

### 1.1 测试对象

本测试用例适用于直播竞拍全栈课程实践系统，主要覆盖：

- 商家 / 主播端：商品发布、规则配置、竞拍管理、异常取消、订单查看。
- 用户端：进入直播间、查看竞拍信息、参与出价、查看排名、接收提醒、查看成交结果。
- 后端服务：REST API、WebSocket 服务、竞拍状态机、出价校验、订单生成。
- 数据层：MySQL / PostgreSQL、Redis 缓存、排行榜数据、出价记录、订单数据。
- 实时通信层：WebSocket 长连接、房间隔离、消息广播、断线重连。

### 1.2 核心测试目标

本次测试重点围绕以下六个方向循序渐进展开：

1. 竞拍规则正确性  
2. 并发出价一致性  
3. WebSocket 实时同步  
4. 断线重连和异常恢复  
5. 越权和非法请求  
6. 性能压测和瓶颈定位  

### 1.3 测试原则

- 先测单用户、单流程，再测多用户、多流程。
- 先测正常路径，再测异常路径。
- 先测功能正确性，再测并发一致性。
- 先测小并发，再逐步提升到中高并发。
- 所有关键业务规则必须以后端校验结果为准，不能只依赖前端限制。
- 测试结果应尽量保留日志、截图、录屏、数据库记录和压测报告。

### TC-RULE-008：超过封顶价的出价应被处理为非法或按封顶价成交

| 项目 | 内容 |
|---|---|
| 用例编号 | TC-RULE-008 |
| 优先级 | 高 |
| 测试阶段 | 边界规则测试 |
| 测试目标 | 验证超过封顶价的出价处理逻辑是否明确 |
| 前置条件 | 当前价为 900，加价幅度为 100，封顶价为 1000，竞拍状态为进行中 |
| 测试数据 | user_001 出价 1100 |

#### 测试步骤

1. 使用接口工具直接提交 1100 元出价。
2. 观察接口返回结果。
3. 查看当前价。
4. 查看竞拍状态。
5. 查看订单记录。

#### 预期结果

以下两种设计均可接受，但系统必须保持一致：

- 方案 A：超过封顶价的出价被拒绝，提示不得超过封顶价。
- 方案 B：超过封顶价的出价按封顶价 1000 处理，并自动成交。

无论采用哪种方案，都必须满足：

- 不允许最终价格超过封顶价。
- 不允许生成多笔订单。
- 用户端和主播端状态一致。
- 数据库状态一致。

#### 可能暴露的问题

- 当前价被更新为超过封顶价的金额。
- 前端显示 1000，数据库记录 1100。
- 订单金额与成交金额不一致。
- 规则定义不清，导致前后端行为不一致。

### TC-RULE-010：连续多次最后时刻出价的延时处理

| 测试目标 | 验证最后时刻连续出价时延时逻辑是否稳定 |
| 前置条件 | 竞拍进行中，结束前 10 秒内出价自动延时 20 秒 |
| 测试数据 | user_001、user_002、user_003 分别在最后 8 秒、最后 6 秒、最后 5 秒出价 |

#### 测试步骤

1. 启动竞拍。
2. 在倒计时进入最后 10 秒后，user_001 出价。
3. 等待倒计时再次进入最后 10 秒后，user_002 出价。
4. 再次等待倒计时进入最后 10 秒后，user_003 出价。
5. 观察每次延时是否生效。
6. 查看最终结束时间。
7. 查看所有用户端倒计时是否一致。

#### 预期结果

- 每次合法出价都按照规则触发延时，或按照系统定义的延时上限处理。
- 如果系统设置了最大延时次数或最大结束时间，应严格按上限执行。
- 延时规则在前端、后端、数据库中保持一致。
- 不出现倒计时突然回退或提前结束。

#### 可能暴露的问题

- 延时重复叠加导致竞拍无限延长。
- 并发延时导致结束时间被覆盖。
- 倒计时出现负数。
- 不同客户端倒计时不一致。

- 服务端记录连接失败原因。

#### 可能暴露的问题

- 任意连接都能接收房间消息。
- WebSocket 只校验 roomId，不校验用户身份。
- 过期 token 仍可长期连接。
- 连接断开后服务端未清理用户状态。
