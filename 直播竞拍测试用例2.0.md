# 直播竞拍测试用例 2.0

> 版本说明：本文档是在第一阶段测试与最新执行结果基础上补充的第二阶段测试用例。第一阶段已经完成核心 smoke、浏览器 E2E、P1/P2 修复验证、安全鉴权、基础性能和 1000 WebSocket 连接测试。本版本重点补齐：10 分钟稳定性、真实混合流量、Redis/MySQL 故障注入、1000 WebSocket 广播延迟、数据一致性专项。  
> 原 30 分钟长时间稳定性测试先降级为 10 分钟，待 10 分钟稳定后再扩展到 30 分钟或更长时间。

---

## 1. 测试背景与目标

### 1.1 当前已完成测试情况

当前系统已经完成并通过：

- 核心 smoke 测试。
- 浏览器 E2E 测试。
- P1 / P2 修复验证。
- 安全鉴权 `TC-SEC-001` ~ `TC-SEC-008`。
- 性能方向 `TC-PERF-001` ~ `TC-PERF-007`。
- `TC-PERF-004` 1000 WebSocket 连接测试。

已经修复的问题包括：

- 未开始竞拍规则编辑缺失。
- 连续延时规则失效。
- 缺少被超越提醒。
- 房间隔离问题。
- 非法金额校验不足。
- 高频出价限流不足。
- 登录 / 角色鉴权缺失。

### 1.2 当前仍需补充验证的问题

- 系统是否可以持续稳定运行，而不仅是短时间压测通过。
- 真实直播竞拍场景下，观看、出价、重连、延时、广播是否稳定。
- 1000 WebSocket 连接是否只是能连接，还是能持续在线并稳定接收广播。
- Redis 或 MySQL 异常时，是否会产生脏数据、重复订单或错误成交。
- 并发、延时、成交、取消、故障恢复后，各层数据是否一致。

### 1.3 第二阶段测试目标

- 从“基础功能正确”提升到“真实场景稳定”。
- 从“接口压测通过”提升到“长连接、广播、混合流量可验证”。
- 从“正常路径可用”提升到“异常情况下不产生错误数据”。
- 从“局部测试结果通过”提升到“Redis、MySQL、HTTP、WebSocket、前端状态一致”。

## TC-STAB-001：100 WebSocket 在线 + 10 分钟持续竞拍稳定性测试

| 项目 | 内容 |
|---|---|
| 用例编号 | TC-STAB-001 |
| 优先级 | 高 |
| 测试目标 | 验证 100 个用户持续在线 10 分钟时，系统连接、广播、出价、状态同步是否稳定 |
| 测试类型 | 长时间稳定性测试 |
| 前置条件 | API、Redis、MySQL、WebSocket 服务正常；已创建一个竞拍中商品 |
| 测试规模 | 100 个 WebSocket 用户，持续 10 分钟 |
| 通过标准 | 服务不崩溃；连接保持率 ≥ 98%；广播到达率 ≥ 98%；无价格回退；无重复订单 |

### 测试步骤

1. 创建一个竞拍商品，设置竞拍时长为 15 分钟。
2. 模拟 100 个用户进入同一直播间，并建立 WebSocket 连接。
3. 所有用户执行 `joinAuction`，接收初始 `auctionSnapshot`。
4. 持续运行 10 分钟。
5. 每 10 秒模拟 1 次合法出价。
6. 每 30 秒统计一次当前在线连接数。
7. 每 30 秒统计一次广播消息到达率。
8. 每 1 分钟查询一次当前价、最高出价用户、排行榜。
9. 测试结束后断开所有连接。
10. 检查服务端连接是否释放。
11. 检查数据库和 Redis 状态是否一致。

### 重点指标

| 指标 | 目标值 |
|---|---|
| WebSocket 连接成功率 | ≥ 99% |
| 10 分钟连接保持率 | ≥ 98% |
| 广播消息到达率 | ≥ 98% |
| 出价接口 P95 | ≤ 800ms |
| WebSocket 广播 P95 | ≤ 1s |
| 价格回退次数 | 0 |
| 重复订单数 | 0 |
| 服务崩溃次数 | 0 |

### 预期结果

- 10 分钟内 WebSocket 连接基本稳定。
- 出价成功后，所有在线用户能收到价格更新。
- 排行榜持续正确更新。
- 当前价不会回退。
- Redis 当前价、数据库当前价、排行榜第一名一致。
- 测试结束后服务端连接数回落，不出现连接残留。

### 可能暴露的问题

- WebSocket 连接逐渐掉线。
- 服务端未清理断开的 socket。
- 内存持续上涨。
- 广播延迟越来越高。
- 排行榜和当前价不同步。
- Redis key 没有过期或持续膨胀。

## TC-STAB-002：500 WebSocket 在线 + 10 分钟持续广播稳定性测试

| 项目 | 内容 |
|---|---|
| 用例编号 | TC-STAB-002 |
| 优先级 | 高 |
| 测试目标 | 验证 500 在线用户下，系统持续广播和连接保持能力 |
| 测试类型 | 中规模稳定性测试 |
| 前置条件 | `TC-STAB-001` 通过；服务端资源监控开启 |
| 测试规模 | 500 个 WebSocket 用户，持续 10 分钟 |
| 通过标准 | 连接保持率 ≥ 95%；广播到达率 ≥ 95%；P95 广播延迟 ≤ 2s |

### 测试步骤

1. 创建一个 15 分钟竞拍商品。
2. 模拟 500 个用户连接同一直播间。
3. 每 5 秒由服务端或测试脚本触发一次竞拍状态广播。
4. 每 15 秒模拟 1 次合法出价。
5. 每分钟记录连接数、广播到达率、广播延迟、CPU、内存。
6. 第 5 分钟随机断开 50 个用户，再让其重新连接。
7. 检查重连后是否能收到最新 `auctionSnapshot`。
8. 10 分钟结束后断开所有用户。
9. 检查连接释放情况和最终数据一致性。

### 重点指标

| 指标 | 目标值 |
|---|---|
| 初始连接成功率 | ≥ 98% |
| 10 分钟连接保持率 | ≥ 95% |
| 广播到达率 | ≥ 95% |
| P95 广播延迟 | ≤ 2s |
| 重连成功率 | ≥ 95% |
| 服务崩溃次数 | 0 |
| 数据不一致次数 | 0 |

### 预期结果

- 500 用户连接后系统不崩溃。
- 广播消息能稳定到达大部分客户端。
- 随机断线用户重连后能恢复最新状态。
- 连接断开后服务端能释放连接。
- 当前价、排行榜、成交状态一致。

### 可能暴露的问题

- 广播循环阻塞。
- 单进程 WebSocket 服务瓶颈明显。
- 部分 socket 长时间无响应。
- 重连后重复订阅，导致一条消息收到多次。
- 房间用户列表未及时清理。

|---|---|
| 用例编号 | TC-STAB-003 |
| 优先级 | 高 |

| 测试类型 | 资源回收测试 |
| 前置条件 | 已完成 `TC-STAB-001` 或 `TC-STAB-002` |
| 通过标准 | 压测结束后连接数回落；无明显内存泄漏；临时 key 可释放 |

### 测试步骤

1. 记录压测前服务端内存、连接数、Redis key 数量。
2. 执行 10 分钟稳定性测试。
3. 测试结束后断开所有 WebSocket 用户。
4. 等待 1-3 分钟。
5. 再次记录服务端内存、连接数、Redis key 数量。
6. 检查房间用户列表是否清空。
7. 检查竞拍结束后相关定时任务是否清理。

| 指标 | 目标 |
|---|---|
| WebSocket 活跃连接数 | 应回落到接近 0 |

| 定时任务 | 不应重复残留 |

### 预期结果

- 所有模拟用户断开后，服务端连接数下降。
- 不出现大量僵尸连接。
- Redis 中临时房间状态不会无限增长。
- 服务运行状态恢复正常。

## TC-MIX-001：100 用户真实混合竞拍流量测试

|---|---|

8. 第 9 分钟提高出价频率，模拟最后时刻抢拍。

10. 等竞拍结束后检查成交结果和数据一致性。

### 重点指标

| 指标 | 目标 |
|---|---|
| WebSocket 连接成功率 | ≥ 99% |
| 出价接口错误率 | 业务失败允许，系统 5xx ≤ 1% |
| 当前价回退次数 | 0 |
| 成交订单数 | 1 |
| 最高出价用户唯一 | 是 |
| Redis/MySQL 最终一致 | 是 |
| 重连恢复成功率 | ≥ 95% |
| 广播 P95 延迟 | ≤ 1s |

### 预期结果

- 只观看用户持续收到竞拍状态更新。
- 低频和高频用户都能按规则出价。
- 出价失败原因明确，例如“价格已变化”“当前用户已领先”“请求过于频繁”。
- 断线用户重连后能拿到最新快照。

- 当前价、排行榜、订单用户一致。

### 可能暴露的问题

- 高频用户影响普通用户出价体验。
- 失败请求没有明确区分业务失败和系统错误。
- 重连用户看到旧价格。
- 最后时刻延时未正确触发。
- WebSocket 广播过多导致客户端卡顿。

## 4. 测试方向三：Redis / MySQL 故障注入测试

> 建议在独立测试环境执行，不要在演示环境随便停止服务。目标不是要求系统在故障时完全可用，而是验证异常情况下不会产生脏数据、重复订单、错误成交。

---

## TC-FAULT-001：Redis 短暂不可用时出价处理

| 项目 | 内容 |
|---|---|
| 用例编号 | TC-FAULT-001 |
| 优先级 | 高 |
| 测试目标 | 验证 Redis 不可用时，系统是否能避免错误价格和错误成交 |
| 测试类型 | 故障注入测试 |
| 前置条件 | 测试环境允许停止 Redis；竞拍进行中 |
| 故障方式 | 停止 Redis 5-10 秒 |
| 通过标准 | 可以拒绝出价，但不能产生错误价格、重复订单或缓存数据库不一致 |

### 测试步骤

1. 创建一个竞拍商品并启动竞拍。
2. 当前价设置为 100。
3. 停止 Redis 服务，或断开 Redis 连接。
4. `user_001` 尝试出价 110。
5. `user_002` 尝试出价 120。
6. 观察接口返回。
7. 恢复 Redis 服务。
8. 查询数据库当前价、出价记录、订单。
9. 查询 Redis 恢复后的当前价和排行榜。
10. 再执行一次正常出价，验证系统是否恢复。

### 预期结果

可接受结果有两种：

- 方案 A：Redis 故障期间系统拒绝出价，返回“服务繁忙”或类似错误。
- 方案 B：Redis 故障期间系统降级到数据库处理，但仍保证一致性。

无论采用哪种方案，都必须满足：

- 不出现价格回退。
- 不生成重复订单。
- Redis 恢复后不会用旧缓存覆盖数据库新状态。
- 错误日志中能看到 Redis 异常。
- Redis 恢复后系统可继续竞拍。

### 可能暴露的问题

- Redis 异常导致接口卡死。
- Redis 写失败但接口返回成功。
- 数据库成功但 Redis 恢复后覆盖新数据。
- 排行榜丢失。
- 服务直接崩溃。

## 11. 阶段性结论

本 2.0 测试用例的重点是将系统测试从“基础功能和短时间性能验证”推进到“真实竞拍场景、长连接稳定性、故障恢复和数据一致性验证”。

如果本阶段测试能够通过，说明系统已经具备以下能力：

- 能稳定支撑一定规模用户持续在线。
- 能处理真实直播竞拍中的观看、出价、重连和最后时刻抢拍。
- 能在 1000 WebSocket 在线下持续广播。
- 能在 Redis / MySQL 异常时避免错误成交和脏数据。
- 能保证 Redis、MySQL、HTTP、WebSocket、前端展示最终一致。

这将比单纯展示“1000 连接成功”更有说服力，也更符合直播竞拍系统对高并发、实时性、稳定性和一致性的核心要求。
