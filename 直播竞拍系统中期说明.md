# 直播竞拍系统中期说明

## 1. 课题背景

本课题围绕“抖音电商直播竞拍”场景，设计并实现一个支持主播发起竞拍、用户实时出价、系统自动校验和成交结算的全栈系统。

当前阶段的重点不是简单做一个页面 Demo，而是验证一个实时竞拍系统在真实业务约束下能否稳定运行：

- 多用户同时在线时，出价状态能否实时同步。
- 高并发出价时，是否会出现非法低价、重复请求、出价丢失、重复订单。
- 竞拍结束、封顶成交、主播取消、故障恢复等边界场景是否一致。
- Redis、MySQL、WebSocket、HTTP 接口和前端展示是否能保持同一份状态。

当前系统已经完成从 P0 MVP 到 P1 演示体验和专项验证的主要闭环，可以用于中期成果展示。

## 2. 当前完成度概览

## 3. 技术栈与系统架构

### 3.1 技术栈

| 层级 | 技术 |
|---|---|
| 主播后台 | React、TypeScript、Vite |
| 用户 H5 | React、TypeScript、Vite |
| 服务端 | Node.js、NestJS |
| 实时通信 | Socket.IO |
| 热点状态 | Redis、Lua、ZSET |
| 持久化 | MySQL、Prisma |
| 本地基础设施 | Docker Compose |
| 自动化验证 | Node.js 脚本、Jest、浏览器 E2E、专项压测脚本 |

### 3.2 架构说明

系统采用前后端分离架构：

- 主播端和用户端分别是两个 React 应用。
- 服务端使用 NestJS 提供 REST API 和 Socket.IO Gateway。
- MySQL 保存商品、直播间、竞拍、出价流水、订单等持久化数据。
- Redis 保存竞拍运行时热点状态、幂等键和排行榜。
- Redis Lua 脚本保证出价处理的原子性。

核心链路如下：

```text
主播端 / 用户端
      |
      | REST API / WebSocket
      v
NestJS API Server
      |
      |-- MySQL + Prisma：持久化商品、竞拍、出价、订单
      |
      |-- Redis + Lua：原子出价、状态推进、排行榜、幂等
      |
      |-- Socket.IO：向直播间用户广播出价、延时、成交、取消
```

## 4. 已实现的核心功能

### 4.1 主播端功能

主播端目前支持：

- 创建商品、直播间和竞拍。
- 配置起拍价、固定加价幅度、封顶价、竞拍时长、自动延时窗口和延时时长。
- 编辑未开始竞拍的规则。
- 开始竞拍和取消进行中的竞拍。
- 分页查看竞拍记录，每页 10 条。
- 点击竞拍记录查看竞拍详情、出价流水、排行榜、订单和状态。
- 管理多件拍品记录，已结束、已取消、进行中的竞拍都有记录可查。

### 4.2 用户端功能

用户端目前支持：

- 创建或选择演示用户。
- 通过竞拍 ID 加入竞拍。
- 查看拍品、当前价、最低可出价、封顶价、倒计时和实时排行榜。
- 提交合法出价。
- 接收新出价、被超越、自动延时、成交和取消提示。
- 断线重连后恢复最新状态。
- 获胜后查看订单并完成模拟支付。

### 4.3 服务端核心能力

服务端目前支持：

- 登录和角色鉴权。
- 出价规则校验：低价、不符合加价幅度、同一领先者连续加价、成交后出价、取消后出价都会被拒绝。
- `requestId` 幂等处理。
- Redis Lua 原子竞价。
- Redis ZSET 实时排行榜。
- 自动延时。
- 封顶价立即成交。
- 倒计时结束自动结算。
- 订单唯一约束，防止重复订单。
- MySQL 短暂不可用时返回 `503 服务繁忙，请稍后重试`。
- Redis / MySQL / 订单生成异常时避免产生脏数据或虚假成交。

## 5. 数据模型

当前主要数据模型包括：

- `User`：用户，区分 `ADMIN` 和 `BIDDER`。
- `Product`：拍品信息。
- `LiveRoom`：直播间。
- `Auction`：竞拍主表，保存状态、规则、当前价、领先者、结束时间等。
- `Bid`：出价流水，使用 `requestId` 唯一约束保证幂等。
- `Order`：成交订单，使用 `auctionId` 唯一约束保证一场竞拍最多一笔订单。

竞拍状态包括：

- `DRAFT`：未开始。
- `RUNNING`：进行中。
- `SOLD`：已成交。
- `UNSOLD`：流拍。
- `CANCELLED`：已取消。

## 7. 中期阶段修复的问题

### 7.1 MySQL 暂时不可用导致 API 不可恢复

问题：

- MySQL 停止后，API 进程曾出现不可用。
- MySQL 恢复后，API 仍无法继续处理正常出价。

修复：

- 增加 Prisma / MySQL 不可用错误识别。
- 通过全局异常过滤器将 MySQL 暂时不可用映射为 `503 服务繁忙，请稍后重试`。
- 避免后台结算任务异常导致 API 崩溃。

结果：

- `TC-FAULT-002` 修复后通过。
- 故障期间不推进 Redis，不产生 MySQL 脏数据。
- MySQL 恢复后可以继续正常出价。

### 7.2 成交订单生成失败导致状态不一致

问题：

- 订单生成失败时，可能出现 Redis 显示 `SOLD`，MySQL 仍为 `RUNNING`。
- 前端可能看到虚假成交。

修复：

- 增加 Redis 状态恢复逻辑。
- 当 Redis 已接受出价但 MySQL 事务失败时，从数据库恢复 Redis 状态和排行榜。
- 测试注入订单生成失败场景，验证不会出现 SOLD 无订单。

结果：

- `TC-FAULT-004` 修复后通过。
- 订单生成失败时不产生虚假成交。
- Redis / MySQL / HTTP 保持一致。

### 7.3 竞拍记录和并行场次展示

问题：

- 主播端竞拍记录过长。
- 创建新竞拍后旧竞拍不够清晰。
- 需要支持查看每件拍品详情。

修复：

- 主播端竞拍列表分页，每页 10 条。
- 按时间排序展示。
- 支持点击竞拍记录查看详情。
- 详情中展示出价流水、排行榜、订单和状态。

结果：

- 主播可以管理多场竞拍。
- 已结束、已取消、进行中的竞拍都有记录可查。

当前阶段仍有明确边界：

- 暂未接入真实直播推流，直播画面为占位展示。
- 暂未接入真实支付，支付为模拟支付。

- Redis 故障提示仍可进一步优化为更稳定明确的 `503 Service Unavailable / 服务繁忙，请稍后重试`。

## 10. AI 功能定位

当前系统的核心链路不依赖外部 AI API。

这点是有意设计的：

- 实时出价、成交、订单生成不能依赖外部大模型接口。
- AI API 延迟、失败或额度问题不应影响竞拍结果。
- API Key 禁止上传到 GitHub 等公共平台，只能通过环境变量管理。

后续可作为 P2 亮点加入 AI 辅助能力：

- 面向商家：拍品文案优化、起拍价/加价幅度/封顶价建议、成交复盘报告。
- 面向买家：拍品估值参考、保守/激进出价建议、直播间热度和历史出价趋势提醒。

第一版 AI 能力可以先不接外部模型，使用规则和统计指标生成可解释建议；后续如接入大模型，也必须作为旁路增强，不能进入实时竞拍关键路径。
